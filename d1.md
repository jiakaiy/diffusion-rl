# 第一部分：按代码文件逐项回答“你要搞懂的问题”

## 1. sft_train.py：SFT 入口脚本

这个文件回答的是：d1 的 SFT 阶段是不是一条独立 pipeline，以及它和 RL 阶段是否共享抽象。

<p class="green">小问 1：d1 的 SFT 是不是独立的一条 pipeline?</p>

是。代码结构上，SFT 从命令行参数开始，经过 load_model_and_tokenizer、load_data、train_model，最后直接构造 dLLMTrainer 并调用 trainer.train()。这条路径没有进入 diffu-grpo 目录，也没有调用 RL trainer。

```text
parse_args -> load_model_and_tokenizer -> load_data -> dLLMTrainer -> trainer.train()
```

所以从 codebase 角度看，SFT 是一个完整但独立的训练入口，而不是一个统一 post-training pipeline 中的一个 stage。

<p class="green">小问 2：它和 RL trainer 有没有共享抽象?</p>

几乎没有真正共享。SFT 侧使用 transformers.Trainer 的子类 dLLMTrainer；RL 侧使用 TRL 的 GRPOTrainer 子类 DiffuGRPOTrainer。二者都加载 LLaDA-style model，也都使用 LoRA，但这只是工程选择相似，不是共享了统一的 model adapter、data adapter 或 trainer interface。

更具体地说，SFT 的 diffusion 特性主要写在 dLLMDataCollator 和 dLLMTrainer.compute_loss 中；RL 的 diffusion 特性写在 DiffuGRPOTrainer.generate、forward_process、_get_per_token_logps 和 _generate_and_score_completions 中。它们不是同一个抽象层的两个配置。

<p class="green">小问 3：SFT 是不是“一个单独脚本”，而不是 unified post-training pipeline 的一部分?</p>

是。sft_train.py 有自己的参数、自己的数据加载、自己的 TrainingArguments、自己的 Trainer 初始化。RL 的 diffu_grpo_train.py 又有另一套参数解析、数据集选择、reward 选择、模型加载和 trainer 初始化。

checkpoint 如何从 SFT 接到 RL，更多是外部运行流程上的约定，而不是由代码中的统一 pipeline 管理。

<p class="green">对应写作部分怎么用</p>

- Codebases as dLLM RL infrastructure：d1 不是只有单个函数，而是包含 SFT 和 RL 两条训练线；但它们是并列工具，不是统一系统。
- A layered open-source ecosystem：d1 位于 “RL adapter / method-level post-training” 层，而不是更高层 framework。
- Representative artifacts：SFT 侧可列出 dLLMTrainer、dLLMDataCollator、masked SFT preprocessing。
- Reproducibility and fair comparison：SFT 与 RL 分开跑，意味着 SFT checkpoint、max_length、LoRA 配置、训练步数等需要单独报告。
- Infrastructure gaps：缺少一个统一的 post-training pipeline 或 stage manager，这是 d1 的明显 gap。

## 2. sft_trainer.py：SFT 的 diffusion loss、forward noising 和数据整理

这个文件回答的是：SFT 阶段到底 diffusion-specific 在哪里，以及 prompt/response 边界如何实现。

<p class="green">小问 1：d1 的 SFT 到底是不是 diffusion-native?</p>

部分是，部分不是。它的训练骨架仍然是 Hugging Face Trainer，但 loss 和 collator 是 diffusion-specific。dLLMTrainer.compute_loss 不是 AR next-token loss，而是对被 mask 的 token 计算 cross entropy，再按 t 做加权；dLLMDataCollator.forward_process 会随机采样 t，并以对应概率把输入 token 替换成 mask token。

所以更准确的说法是：SFT 的目标函数和数据扰动是 diffusion-native，但基础训练框架是传统 Trainer patch。

<p class="green">小问 2：prompt / response 的分界是怎么实现的?</p>

分界靠 preprocess_dataset 里保存 prompt_lengths，再由 dLLMDataCollator.__call__ 使用。具体逻辑是：先把完整 prompt+response tokenized 成 input_ids；然后保存 prompt 部分长度；collator 做 forward noising 后，会把 prompt 区域恢复成原 token，并把 prompt 区域 labels 设成 -100，这样 prompt 不参与 loss。

这说明 SFT 实际学的是 completion-only masked reconstruction：给定完整 prompt，只恢复 response 中被 mask 的 token。

<p class="green">小问 3：这套 SFT 实现是否容易复用到别的 dataset / model?</p>

复用性有限。preprocess_dataset 明确依赖 s1K-style 字段，例如 question、thinking_trajectories[0] 和 attempt。也就是说，数据格式假设被写进 preprocessing 里。模型方面，mask_token_id 可以通过 tokenizer 或显式参数传入，但 sft_train.py 中 data_collator 又直接传了 mask_token_id=126336，这明显偏向 LLaDA。

所以它可以作为 masked SFT 的参考实现，但还不是一个通用 SFT data adapter。

<p class="green">对应写作部分怎么用</p>

- Codebases as dLLM RL infrastructure：SFT 侧说明 codebase 已经不只是算法伪代码，而是实现了可运行的 diffusion post-training stage。
- Layered ecosystem：属于 model-specific post-training patch，而非通用 SFT framework。
- Representative artifacts：dLLMDataCollator.forward_process 和 dLLMTrainer.compute_loss 是最有代表性的 SFT 组件。
- Reproducibility：需要报告 prompt formatting、mask schedule、max_length、PAD/token handling 和 SFT 数据字段。
- Infrastructure gaps：缺少数据格式抽象；SFT preprocessing 和 s1K 格式耦合；mask_id 也带有模型特定性。

## 3. diffu_grpo_train.py：RL 训练入口和任务分支

这个文件回答的是：d1 的 RL pipeline 是怎么启动的，以及 reward / dataset 是否被统一抽象。

<p class="green">小问 1：这个 repo 的 RL 部分是不是统一 trainer + 多个 config?</p>

表面上是。diffu_grpo_train.py 统一实例化 DiffuGRPOTrainer，并使用 DiffuGRPOConfig / ModelConfig 来控制训练。但是数据集和 reward function 的选择仍然通过 if/elif 按 dataset 名称手动分支。

所以它是 “一个 trainer + 多个手写任务分支”，不是完全 registry-style 或 plugin-style framework。

<p class="green">小问 2：数据和 reward 是不是还 heavily task-specific?</p>

是。代码中 gsm8k、countdown、sudoku、math、code 分别走不同 dataset loader 和 reward function 组合。GSM8K 用 XML/format/int/correctness reward；Countdown 用 countdown_reward_func；Sudoku 用 sudoku_reward_func；code 用 xmlcount_reward_func + coding_reward_func。

这说明 reward 和 data 不是从统一任务 schema 自动构造，而是每个 benchmark 单独接线。

<p class="green">小问 3：这是不是 end-to-end，但 abstraction 不够强的典型例子?</p>

是。它确实能把 data、reward、model、LoRA、trainer 串起来训练，所以是一个可运行的 end-to-end RL 脚本。但它的 abstraction 不强，因为任务差异都暴露在 main() 的分支里。增加新任务大概率要新增 loader、reward function，并改 main()。

<p class="green">对应写作部分怎么用</p>

- Codebases as infrastructure：说明 d1 repo 已经承担实验 pipeline 的作用，不只是 paper appendix 代码。
- Layered ecosystem：d1 是 RL adapter 层，复用了底层模型和 Trainer/TRL 基础设施。
- Representative artifacts：DiffuGRPOTrainer 是核心 artifact，train.py 是把 reward/data 接入 trainer 的 orchestration。
- Reproducibility：dataset 分支、reward 组合、seed、LoRA、4-bit quantization 都影响复现。
- Infrastructure gaps：缺少 task registry、reward registry 和统一 task interface。

## 4. diffu_grpo_config.py：哪些设计被参数化，哪些被写死

这个文件回答的是：d1 到底把哪些实验条件暴露成 config，以及哪些假设仍然绑定 LLaDA / 特定 benchmark。

<p class="green">小问 1：哪些 design choice 被参数化了?</p>

不少关键设置已经被参数化，包括 max_prompt_length、num_generations、max_completion_length、temperature、learning_rate、beta、num_iterations、epsilon、reward_weights、generation_batch_size、block_length、diffusion_steps、cfg_scale、remasking、dataset、p_mask_prompt、random_masking 等。

这说明 d1 并不是完全硬编码实验；它已经把很多训练和生成条件放进 config 层。

<p class="green">小问 2：哪些东西仍然写死或高度模型特定?</p>

最明显的是 mask_id 默认值 126336，并且 metadata 直接写明 default is from LLaDA。dataset 也只是一个字符串字段，实际支持哪些任务由 diffu_grpo_train.py 的 if/elif 决定。block_length、diffusion_steps、max_completion_length 虽然可配置，但默认组合也隐含了 LLaDA-style fixed-length diffusion generation。

<p class="green">小问 3：fair comparison 时，哪些超参数最容易导致结果不可比?</p>

至少要报告：base model / model_path、max_completion_length、diffusion_steps、block_length、remasking、temperature、p_mask_prompt、random_masking、num_iterations、num_generations、beta、epsilon、reward weights、LoRA 配置、dataset split 和 checkpoint selection。

这些设置不只是工程细节。对 dLLM 来说，generation length、diffusion steps 和 remasking 直接改变 rollout 分布；p_mask_prompt 和 random_masking 直接改变 log-prob estimator；reward weights 改变优化目标。

<p class="green">对应写作部分怎么用</p>

- Codebases as infrastructure：config 决定实验对象是什么，因此 codebase 不只是实现算法，也定义实验条件。
- Layered ecosystem：d1 暴露了一些 trainer/generation knobs，但不是统一的 framework config schema。
- Representative artifacts：DiffuGRPOConfig 可作为 d1 的配置 artifact。
- Reproducibility：这一节最直接服务于 fair comparison，需要强调必须报告这些 hyperparameters。
- Infrastructure gaps：仍有 LLaDA-specific defaults 和任务集合硬编码，说明模型/任务 adapter 不充分。

## 5. diffu_grpo_trainer.py：d1 的核心 trainer、rollout、estimator 和 reward 汇合点

这是最重要的文件。它回答的是：d1 为什么属于 Family A，以及为什么说 rollout、likelihood estimation、reward、optimization 被绑在同一个 trainer 里。

<p class="green">小问 1：d1 的 trainer 为什么是典型 Family A?</p>

因为 DiffuGRPOTrainer 直接继承 TRL 的 GRPOTrainer。compute_loss 中仍然保留 GRPO/PPO 风格的 old policy log-prob、current policy log-prob、ratio、clip、advantage、KL penalty。真正被替换的是概率计算方式：AR 的 token log-prob 被 _get_per_token_logps 这个 diffusion-specific estimator 替代。

所以 d1 不是 diffusion-native RL framework，而是 “GRPO trainer + diffusion generation/log-prob patch”。

<p class="green">小问 2：generation、likelihood estimation、reward、advantage 为什么都绑在 trainer 里?</p>

_generate_and_score_completions 直接完成 prompt tokenization、diffusion generation、completion 截取、EOS mask、old/ref log-prob 预计算、decode、reward function 调用、reward gather、group mean reward 和 advantage 计算。_prepare_inputs 又控制何时重新生成、何时复用 buffered inputs。

这说明 trainer 不只是优化器。它同时扮演 rollout engine、estimator driver、reward aggregator 和 GRPO optimizer。

<p class="green">小问 3：one-step estimator 在代码里怎么实现?</p>

核心在 _get_per_token_logps。它先根据 logits_to_keep 划分 prompt 与 completion；然后对每个 mask_seed 调 forward_process。forward_process 会随机 mask prompt token，但 completion token 全部 mask。随后把所有 perturbed sequences 拼成一个 batch，一次前向得到 logits，只取 completion 部分 logits，并用 cross entropy 对实际 completion token 取负，得到 per-token log-prob estimate。

这正是 paper 中的 “prompt random masking + one-step completion reconstruction”。

<p class="green">小问 4：prompt masking regularization 为什么能减少 online generations?</p>

_prepare_inputs 只有在 global_step % num_iterations == 0 时重新调用 _generate_and_score_completions；否则复用 _buffered_inputs。也就是说，同一批 rollout completion 会被多个 inner updates 重复使用。为了避免每次看到完全相同的训练视角，_generate_and_score_completions 生成多个 mask_seeds；compute_loss 每次 inner update 取对应 seed，用不同 prompt mask pattern 重新估计 log-prob。

所以 random prompt masking 的代码作用是：让同一批 completion 在多个 inner updates 中产生不同 estimator views，从而提高样本利用率，减少昂贵的 diffusion online generation。

<p class="green">小问 5：这个 trainer 最大的基础设施缺陷是什么?</p>

第一，没有显式 trajectory object。generate 内部走了多步 denoising，但最终只返回完整 token sequence，trainer 后续只处理 final completion。第二，没有 standalone likelihood estimator interface，_get_per_token_logps 是 trainer 方法。第三，reward 和 grouped advantage 也在 trainer 内完成。第四，代码里还有 LLaDA-specific 默认 mask_id，以及 generation 中 attention mask 未真正使用的 warning。

<p class="green">对应写作部分怎么用</p>

- Codebases as infrastructure：这个文件最能说明 d1 的 repo 如何把生成、打分和更新组合成一个可运行的 RL loop。
- Layered ecosystem：d1 是 AR/TRL trainer 之上的 method adapter，不是 framework 层。
- Representative artifacts：DiffuGRPOTrainer、generate、_get_per_token_logps、_prepare_inputs 是 d1 最核心 artifacts。
- Reproducibility：generation settings、random_masking、num_iterations、old/ref log-prob 预计算和 completion_mask 都是实验协议的一部分。
- Infrastructure gaps：trajectory、estimator、rollout、reward 都没有被充分解耦。

## 6. reward_func.py：reward / verifier 层

这个文件回答的是：d1 的 reward 是否模块化，以及它对公平比较有什么影响。

<p class="green">小问 1：reward 是不是统一接口?</p>

接口形式上比较统一：reward 函数大多接收 prompts、completions 和任务相关 kwargs，然后返回 float list。但内部实现高度任务特定。GSM8K、MATH、Countdown、Sudoku、coding 各自有不同 parser、格式检查和 correctness 逻辑。

<p class="green">小问 2：d1 的 reward 设计是不是 heavily benchmark-engineered?</p>

是。GSM8K 有 XML/format/int/correctness 多个 reward；MATH 用 boxed answer 解析；Countdown 要检查表达式是否使用给定数字并等于 target；Sudoku 只对空格位置计算正确比例；coding 要抽取 python code block 并跑 unit tests。

<p class="green">小问 3：coding reward 为什么已经像一个 execution backend?</p>

coding_reward_func 不只是字符串比较。它会抽取代码块、检查代码是否在 answer tag 内、解析测试函数、替换 function name、过滤危险模块、用 multiprocessing 执行测试并设置 timeout 和内存限制。这已经接近一个轻量级 code execution verifier backend。

<p class="green">小问 4：fair comparison 时，不同 reward 组合会带来什么问题?</p>

如果两个 repo 都说在 GSM8K 或 code 上训练，但 reward parser、format reward 权重、correctness 判断、代码执行限制不同，模型实际优化的目标就不相同。尤其 d1 把 format reward 和 correctness reward 加在一起，最终 reward 不只是“答案对不对”。所以比较时不能只看 benchmark 名字，还要看 reward construction。

<p class="green">对应写作部分怎么用</p>

- Codebases as infrastructure：reward_func.py 说明 codebase 已经承担 verifier 的角色。
- Layered ecosystem：这是 evaluation/reward layer 的雏形，但仍然 task-specific。
- Representative artifacts：GSM8K format/correctness reward、Countdown verifier、Sudoku scorer、coding execution reward 都是可列组件。
- Reproducibility：reward parser、format reward 和代码执行环境必须报告，否则比较不公平。
- Infrastructure gaps：缺少真正统一的 verifier API；每个 benchmark 的 reward 逻辑都写成独立函数。

## 7. data_utils.py：任务数据入口和 prompt schema

这个文件回答的是：d1 是否有统一数据接口，以及新任务扩展成本如何。

<p class="green">小问 1：数据接口是不是统一到 “prompt + metadata”？</p>

基本是。各个 loader 都把数据 map 成 prompt 列表加上任务字段。例如 GSM8K 返回 prompt 和 answer；Countdown 返回 prompt、target、numbers；Sudoku 返回 prompt、puzzle、solution；code 返回 prompt 和 answer，其中 answer 包含 solution 和 tests。

<p class="green">小问 2：哪些字段通用，哪些字段任务特有?</p>

通用字段主要是 prompt。除此之外几乎都任务特有：answer、target、numbers、puzzle、solution、tests 等字段都服务于各自 reward function。这种设计能跑，但说明 reward/data 是成对绑定的。

<p class="green">小问 3：repo 能不能轻松扩展到新任务?</p>

可以扩展，但不是 plug-and-play。新任务通常需要新增 dataset loader、定义 prompt template、增加 task-specific metadata、写新的 reward function，并在 diffu_grpo_train.py 的 if/elif 中接入。没有通用 task registry 或 verifier registry。

<p class="green">对应写作部分怎么用</p>

- Codebases as infrastructure：data_utils.py 说明 d1 提供了 benchmark adapters，而不只是 trainer。
- Layered ecosystem：这是 evaluation/benchmark 接入层，但还不是通用任务框架。
- Representative artifacts：get_gsm8k_questions、get_countdown_questions、get_sudoku_questions、get_math_questions、get_code_questions 可列为 dataset adapters。
- Reproducibility：prompt template 和 metadata schema 会影响 reward extraction 和最终结果。
- Infrastructure gaps：缺少统一 task schema 和 registry。

# 第二部分：把上述代码观察映射到五个写作小节

这一部分不是重复文件分析，而是告诉你：如果要写 “Codebases, Frameworks, and Experimental Infrastructure” 中的五个小节，d1 这个 repo 分别能贡献什么材料。

## 1. Codebases as dLLM RL infrastructure

d1 可以用来说明：当前 dLLM RL 代码库已经在实践中承担“实验基础设施”的角色。它不只是实现一个 loss，而是把 SFT 数据处理、diffusion generation、reward evaluation、advantage computation 和 GRPO-style update 都组织起来。

不过，它的基础设施性质是 method-specific 的：SFT 和 RL 是两条独立路径，RL 的核心 pipeline 集中在 DiffuGRPOTrainer 内。可以写成：d1 functions as a runnable method adapter rather than a general post-training platform.

## 2. A layered open-source ecosystem

d1 在 layered ecosystem 里应放在 “RL adapter / method-level post-training adapter” 层。它依赖 LLaDA-style masked dLLM 作为 substrate，依赖 Hugging Face Trainer / TRL GRPOTrainer 作为训练骨架，然后在上面添加 diffusion-specific generation 和 log-prob estimation。

它不是 foundation model repo，也不是 serving system，也不是 full framework。它位于中间层：把已有 foundation model 接到 RLVR 训练上。

## 3. Representative artifacts and reusable components

d1 中最值得列出的 artifacts 是：

- dLLMDataCollator.forward_process：SFT 侧 forward noising。
- dLLMTrainer.compute_loss：masked diffusion SFT loss。
- DiffuGRPOTrainer.generate：diffusion rollout/generation routine。
- _get_per_token_logps：one-step per-token log-prob estimator。
- _prepare_inputs / _generate_and_score_completions：rollout reuse、reward、advantage、old/ref logps 的汇合点。
- reward_func.py：benchmark-specific verifier/reward collection。

这些组件“可参考”，但并不完全 reusable，因为它们大多绑定在 trainer、LLaDA-style mask token 和具体任务字段上。

## 4. Reproducibility protocols and fair comparison

d1 最能提醒我们：比较 dLLM RL 方法不能只写 benchmark 分数。必须报告 generation protocol、diffusion_steps、block_length、max_completion_length、remasking、temperature、p_mask_prompt、random_masking、num_iterations、beta/epsilon、reward composition、dataset split、LoRA/quantization 和 checkpoint handoff。

代码里这些设置分散在 config、train script、trainer 和 reward files 中。它们共同定义实验条件。不同 repo 即使都叫 “GRPO for dLLM”，如果这些协议不同，也不是公平比较。

## 5. Infrastructure gaps

d1 暴露出的 gaps 很清楚：

- 没有显式 trajectory abstraction：多步 denoising 只在 generate 内部发生，trainer 后面只看到 final completion。
- 没有独立 likelihood / ratio estimator interface：_get_per_token_logps 是 trainer 方法。
- 没有独立 rollout engine：generate、buffer、reward、advantage 都在 trainer 内部。
- 没有统一 task/reward adapter：data 和 reward 都按 benchmark 手写。
- 没有统一 SFT/RL stage manager：SFT 和 RL 是分开的脚本。
- 模型特定默认值仍然存在：例如 mask_id=126336。

# 第三部分：d1 这个 repo 最后在五个小节中可以怎么写

下面是可以放进你自己笔记或 Overleaf 草稿的写法骨架。语气参考你们现有 TMLR 草稿：先给判断，再解释代码证据和 caveat，不写成“工具说明书”。

## 1. Codebases as dLLM RL infrastructure：d1 的角色

d1 is a representative example of the first wave of dLLM RL infrastructure: it makes GRPO-style reasoning RL runnable on a masked diffusion language model. The repository is not merely a loss implementation; it contains a full RL training entry point, task-specific dataset loaders, reward functions, diffusion generation code, log-probability estimation, and GRPO-style policy updates. At the same time, its infrastructure remains method-specific. The SFT and RL stages are implemented as separate scripts, and the RL loop is concentrated inside one trainer class rather than decomposed into reusable rollout, reward, estimator, and objective layers.

中文理解：d1 可以写成“第一代可运行 dLLM RL adapter”。它已经是实验基础设施，但不是成熟 framework。

## 2. A layered open-source ecosystem：d1 放在哪一层

Within the layered ecosystem, d1 sits in the RL-adapter layer. It relies on LLaDA-style masked diffusion as the modeling substrate and TRL/Hugging Face training abstractions as the optimization substrate. Its main role is to bridge these layers by patching diffusion-specific rollout and probability estimation into an AR-derived GRPO trainer. This placement is important because it shows that early dLLM RL infrastructure largely adapts autoregressive tooling rather than replacing it with diffusion-native interfaces.

中文理解：d1 不是 foundation 层，不是 serving 层，也不是 Family B 式 framework；它是 AR trainer 到 masked dLLM 的 adapter。

## 3. Representative artifacts and reusable components：d1 可列哪些组件

The most reusable components in d1 are the masked SFT collator and loss, the diffusion generation routine, the one-step per-token log-probability estimator, and the benchmark-specific reward functions. The estimator is the most method-defining component: it replaces AR token log-probabilities with a single-step reconstruction probability computed from a randomly perturbed prompt and a fully masked completion. However, these components are not exposed as standalone modules. They are embedded in the trainer and are therefore easier to reuse by copying code than by composing abstractions.

中文理解：可以列 artifact，但要强调这些 artifact 的 reusable 程度有限。

## 4. Reproducibility protocols and fair comparison：从 d1 看应该报告什么

d1 also illustrates why fair comparison in dLLM RL requires more than reporting benchmark accuracy. The training result depends on the base model, the fixed completion length, diffusion steps, block length, remasking rule, prompt masking probability, number of inner iterations, KL and clipping parameters, reward composition, LoRA configuration, and the exact parsing rules used by each reward function. These settings are distributed across the config, trainer, data loaders, and reward functions. A fair protocol should therefore report both algorithmic choices and systems choices, especially those affecting rollout distribution and reward construction.

中文理解：d1 是一个很好的例子，说明 dLLM RL 的“实验设置”本身就是算法的一部分。

## 5. Infrastructure gaps：d1 暴露了什么缺口

The main infrastructure gap exposed by d1 is the absence of first-class diffusion-specific abstractions. The trainer never stores a denoising trajectory object; it receives final completions and assigns sequence-level rewards to completion tokens. The likelihood estimator is implemented as a trainer method rather than a replaceable estimator interface. Rollout, scoring, old/reference log-probability computation, reward aggregation, and advantage construction all occur inside the same trainer. Data and reward interfaces remain benchmark-specific, and SFT and RL are separate pipelines. These limitations do not reduce d1’s methodological importance, but they show why later systems move toward trajectory-aware training and more framework-like organization.

中文理解：d1 的价值是让事情跑起来；d1 的限制是没有把 trajectory、estimator、rollout、reward 变成通用组件。

# 最后给你的阅读顺序提醒

1. 先读 diffu_grpo_train.py：确认 RL 入口、dataset/reward 分支、model/LoRA/trainer 如何接起来。
2. 再读 diffu_grpo_config.py：标出所有会影响 fair comparison 的参数，尤其 generation 和 masking 相关参数。
